# Phase 3 Account, Secret, Pool, and Quota Implementation Plan

**Goal:** Implement tenant-scoped AJiaSu account inventory, envelope-encrypted credential rotation, Vault/KMS provider adapters, account pools, transactional capacity, tenant quotas, bulk import, API routes, and Phase 3 exit gates.

**Architecture:** New `secrets`, `accounts`, and `pools` modules use Phase 2 transaction, RLS, identity, idempotency, audit, and HTTP contracts. Credentials are opaque encrypted payloads inside the account module; no HTTP read path invokes provider `Open`.

## Execution rules

- Use only fake credentials generated by tests.
- Never log request bodies, credential fields, ciphertext, wrapped DEKs, or external secret references.
- Every tenant repository operation runs through `database.InTenantTx` with a server-derived tenant and actor.
- Handlers call module services and never issue SQL.
- Generated sqlc files are not hand-edited.
- Writes, audit, Outbox, and idempotency results commit atomically.
- Do not add Phase 4 scheduling, Runner, Agent, Redis, node, or endpoint code.
